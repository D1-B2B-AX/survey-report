# 만족도 리포팅 스킬

이 스킬은 만족도조사 raw data를 읽어 기업담당자용·강사용 메일 리포트를 생성합니다.

> **핵심 원칙**: 검증은 코드, 판단은 사람. LLM이 만든 걸 LLM이 검증하지 않습니다.
> **언어 규칙**: 모든 대화, 안내 문구, 블록 내용은 반드시 한국어로 작성합니다. 영어로 전환하지 않습니다.

---

## Phase 0: 환경 확인

**⚠️ 매 실행마다 반드시 check-env.js를 실행합니다.** 이전에 실행한 적이 있더라도, 연속 실행이더라도 건너뛰지 않습니다. 업데이트 확인은 이 스크립트에서만 수행됩니다.

```bash
cd ~/.claude/skills/report && node scripts/check-env.js
```

check-env.js 결과를 확인한 뒤, 아래 항목별로 대응합니다.

**환경 확인 후 안내**: check-env.js가 정상 완료되면, 아래 안내를 대화창에 출력합니다:
> "💡 리포팅 중 명령 실행 승인 팝업이 뜰 수 있습니다. 번거로우시면 `/report_OM사용가이드`의 '권한 설정' 섹션을 참고해주세요."

```
1. Node.js 설치 여부 확인 (node -v)
   → 없으면: winget install OpenJS.NodeJS.LTS 자동 설치 시도
   → 자동 설치 실패 시: "Node.js 설치가 필요합니다. https://nodejs.org 에서 LTS 버전을 설치해주세요." 안내 후 중단
   → 있으면: 다음

2. 필요 패키지 확인 (이 플러그인 폴더의 node_modules 존재 여부)
   → 없으면: 이 플러그인 폴더(~/.claude/skills/report/)에서 npm install 자동 실행
     (패키지: xlsx — 엑셀 파싱)
   → 있으면: 다음

3. 플러그인 업데이트 확인 (git fetch로 원격 최신 버전 비교)
   → 최신이면: 조용히 넘어감
   → 업데이트 있으면: "플러그인 업데이트가 있습니다. 업데이트할까요?" → '예' 시 git pull 실행
   → 네트워크 안 되면: 조용히 넘어감 (현재 버전으로 진행)
   → git 레포가 아니면 (수동 설치 등): 건너뜀
```

**플러그인 시작 타임스탬프 기록**: check-env.js 결과 JSON에 포함된 `currentTime`(ISO), `currentTimeKST`(한국시간), `fileTimestamp`(파일명용 YYYYMMDD_HHMM)을 기록합니다. **LLM은 절대 시각을 추측하지 않고, check-env.js가 출력한 값만 사용합니다.**
```
check-env.js 출력 예시:
  "currentTime": "2026-04-02T07:30:00.000Z",
  "currentTimeKST": "2026-04-02 16:30",
  "fileTimestamp": "20260402_1630"
```
- `currentTime` → meta.json `pluginStartTime`에 저장
- `fileTimestamp` → 이후 모든 파일명의 `{YYYYMMDD}_{HHMM}` 부분에 사용
- `currentTimeKST` → 세션 로그의 시각 표기에 사용

## Phase 1: Input 수집

### 1-0. 필요 파일 안내 + raw data 전달받기

OM에게 **한 번의 안내로** 필요 파일 설명과 raw data 경로 요청을 함께 합니다:

> "만족도 결과보고를 작성할게요. 아래 파일을 전달해주세요.
> 1. **만족도 설문 raw data** (필수)
> 2. **강의관리 시트** (선택 — 있으면 운영진 의견이 더 구체적으로 생성됩니다. 없으면 없이도 진행 가능)
>
> 💡 파일 전달 방법:
> - **파일을 대화창에 드래그**해도 됩니다
> - 또는 파일 경로를 직접 입력: 파일 탐색기에서 파일 클릭 → **Ctrl+Shift+C** → 붙여넣기
>
> 📌 다른 과정의 리포팅을 진행하실 때는 새 터미널에서 Claude Code를 열어주세요. (1과정 1세션)"

OM이 파일을 전달하면 바로 읽습니다. raw data와 강의관리 시트를 한번에 전달받으면 이후 Phase에서 재사용합니다.

**⚠️ Phase 1에서는 파일만 받습니다.** 리포트 범위, 특이사항 등은 Phase 2에서 메타 정보와 함께 한꺼번에 확인합니다.

전달받은 파일 경로에서 직접 읽습니다.
파일명도 메타 정보 추출에 활용하므로 함께 기록합니다.

### 1-2. 강의관리 시트 확인

**⚠️ 이 단계를 건너뛰지 않습니다.** OM이 1-0에서 강의관리 시트를 함께 전달했으면 그대로 사용합니다. 전달하지 않았으면 아래와 같이 요청합니다:

> "**강의관리 시트**도 전달해주세요. 싱크업 문서 내 강의관리 탭도 가능합니다.
> 없으시면 '없음'이라고 알려주세요."

- **있음**: 아래 규칙에 따라 읽기
- **없음**: 운영진 의견은 보편적 수준으로 생성. OM에게 안내:
  "강의관리 시트가 없으므로 운영진 의견은 점수와 주관식 패턴만으로 작성됩니다. 현장 맥락이 반영되지 않아 내용이 일반적일 수 있습니다."

> **🚫 강의관리 시트/싱크업 문서 읽기 — 절대 규칙**
>
> **"강의요약"과 "교육 개요" 2개 탭만 읽습니다. 그 외 탭은 절대 읽지 않습니다.**
> 시트 목록을 확인한 뒤, 이 2개 탭(또는 유사 이름)만 골라서 읽습니다.
> `for (const name of wb.SheetNames)` 같은 전체 시트 순회를 하지 않습니다.
> OM이 "다른 탭도 봐줘"라고 명시적으로 요청한 경우에만 다른 탭을 참조합니다.

**강의관리 시트/싱크업 문서 읽기 순서**:

1. **시트 목록만 먼저 확인** — `wb.SheetNames`를 출력하여 어떤 탭이 있는지 확인합니다. 이 단계에서 탭 내용을 읽지 않습니다.
2. **"강의요약" 탭** (또는 "강의관리" 등 유사 이름)을 찾아서 **이 탭만** 읽습니다. 운영진 의견, 이슈 내용이 들어있는 핵심 탭이므로 전체를 자세히 읽습니다. 차수/직군별로 나뉜 경우 (예: "강의요약(리더직군)") 해당 탭 모두 읽습니다.
3. **"교육 개요" 탭** — `parse-sheet.js --overview`로 자동 읽기:
   ```bash
   cd ~/.claude/skills/report && node scripts/parse-sheet.js "강의관리시트경로" --overview
   ```
   - `found: true`면 → `courseInfo`(과정명, 교육형태, 교육일정, 교육장소, 교육대상)와 `people`(기업담당자, FC담당자, 강사, 조교)이 JSON으로 반환됩니다. 병합 셀을 자동 처리하므로 추가 셀 읽기가 필요 없습니다.
   - `found: false`면 → 교육 개요 탭이 없는 비표준 양식입니다. 아래 키워드로 수동 탐색합니다:
     - "과정명" / "과정" → 과정명
     - "교육 형태" / "교육형태" → 대면/비대면
     - "교육 일정" / "교육일정" / "일정" → 교육 일정
     - "교육 시수" / "시수" → 교육 시수
     - "교육 대상" / "대상" → 교육 대상
     - "강사님" / "강사" / "강사명" → 강사명. 상단 메타 영역뿐 아니라 하단 담당자 목록에도 있을 수 있음.
4. **그 외 탭은 읽지 않습니다.** "운영 표준", "안내 멘트", "체크리스트", "운영 캘린더", "강의사진" 등은 무시합니다.

싱크업 문서를 전달받은 경우에도 동일하게 적용합니다. 탭명이 정확히 일치하지 않을 수 있으므로 (예: "강의관리", "강의 요약", "교육개요" 등 띄어쓰기/워딩 차이), 유사한 이름의 탭을 찾아 적용합니다.

- ⚠️ 주의: 강의관리 시트의 "강의요약"은 raw data의 "요약/통계" 탭과 성격이 다릅니다. 강의관리 시트에서는 오히려 가장 중요한 탭이므로 반드시 참조합니다.

### 1-3. 시트 자동 감지

raw data를 읽을 때 아래 순서로 자동 감지합니다:

```
① 시트 탭 이름 확인
   → 차수/과정 구분 (예: "리더1차수(9.2)", "영업1차수(9.3)")
   → 강사 구분 (탭명에 강사명이 포함된 경우, 예: "홍길동_데일리", "강사A_세션1")
   → 데이터 시트 / 비데이터 시트 분류 (요약, 통계 등은 비데이터로 분류하되 스킵하지 않음 — 필요 시 참조 가능)

② 헤더 행 읽기
   → 문항 텍스트 자동 추출
   → 메타 열 식별 (응답자ID, 타임스탬프 등)

③ 응답값 패턴 분석 + 문항 텍스트 판별
   → 숫자 1~5: 5점 척도 객관식
   → 숫자 1~10: 10점 척도 (NPS/추천지수 등)
   → 양극단 키워드 (난이도 인식, 강의 속도 등): 양극단 척도
   → 문항 텍스트에 주관식 유도 키워드: 주관식 확정
   → 정보성 문항 키워드 (소속, 차수, 적격성 등): info
   → 고유값 적고 반복: 선택형
   → 나머지 자유 텍스트: 주관식
```

자동 감지 결과는 내부적으로 기록하고, Phase 2의 메타 정보 확인 단계에서 OM에게 한꺼번에 확인받습니다. 이 단계에서 별도로 OM에게 질문하지 않습니다.

## Phase 2: 메타 정보 추출 → OM 확인

아래 소스에서 과정 정보를 자동 추출합니다:

```
추출 우선순위:
  1순위: 강의관리 시트 (기업명, 과정명, 차수, 강사명, 일정 등)
  2순위: raw data 파일명 (기업명, 과정명 거의 항상 포함)
  3순위: raw data 시트 내용 (시트 탭명, 문항 내 차수 선택지 등)
  4순위: OM 직접 입력 (위에서 못 찾은 것만)

⚠️ 교육일시는 타임스탬프(설문 응답 시각)로 추정하지 않는다.
   타임스탬프는 교육일이 아니라 응답일이며, 다차수 동일 링크 등에서
   오판 위험이 있다. 확인 안 되면 "(미확인)"으로 표시하고 OM에게 질문한다.
```

추출 결과 + 리포팅 범위 + 특이사항을 **한꺼번에** OM에게 확인받습니다:

> "다음 정보를 확인했어요:
> - 기업명: ○○○
> - 과정명: ○○○
> - 차수: ○○○
> - 강사: ○○○
> - 교육일시: ○○○
> - 응답인원: ○○명 (raw data 기준)
> - 리포트 범위: 기업담당자용 + 강사용
> *(다중 시트/그룹이 감지된 경우 아래 추가)*
> - 데이터 시트: N개 (예: 0221, 0307, 0321) — 전부 리포팅 예정
> - 그룹 구분: A~D그룹 감지됨 — 전체 통합 예정 *(특정 그룹만 원하시면 알려주세요)*
>
> *(마무리 문구 — 상황에 따라 분기)*
> - **다중 시트/그룹 등 확인사항이 있는 경우**: "이외에도 수정할 부분이나 특이사항이 있으면 알려주세요. (예: 강사용만 필요, A그룹만 리포팅, 0221 시트만 등)"
> - **추가 확인사항 없이 메타 정보만 확인하는 경우**: "수정할 부분이나 특이사항이 있으면 알려주시고, 맞으면 '확인'이라고 해주세요."

**⚠️ 다중 시트/그룹 감지 시 반드시 OM에게 확인합니다.** 자동으로 통합 시트를 선택하거나 전체 리포팅으로 진행하지 않습니다. 통합 시트와 날짜별/차수별 시트가 동시에 존재하면, 어떤 시트를 사용할지 OM에게 물어봅니다.

**그룹 구분 열 감지**: 데이터에 "그룹", "소속", "반" 등 그룹 구분 열이 있으면 메타 정보 확인 시 안내합니다. OM이 특정 그룹만 지정하면 해당 그룹 데이터만 필터링하여 사용합니다.

**⚠️ 다중 그룹/다중 시트 블록 검토 기본값**: 그룹 또는 시트가 여러 개인 경우, 블록 검토(객관식/주관식/운영진 의견) 각 sub-tab 안에서 **그룹별(또는 시트별)로 구분하여 순차 배치**하는 것이 기본값입니다. 전체 통합 표를 먼저 보여준 뒤, 그룹별 표를 순서대로 배치합니다. OM이 "통합으로 보여줘"라고 요청하면 그때 합칩니다.

**리포트 범위 분기** (OM 응답에 따라):
- **기본 (기업+강사)**: 기업담당자용 메일 + 강사용 메일 모두 생성
- **강사용만**: 기업담당자용 블록(블록 3)과 메일 미리보기 기업담당자용 탭을 생략
- **기업담당자용만**: 강사용 블록(블록 4, 2차 필터링)과 메일 미리보기 강사용 탭을 생략
- **특정 시트/그룹만**: 해당 데이터만 필터링하여 사용
- 별도 언급이 없으면 기본(기업+강사, 전체 데이터)으로 진행합니다.

OM이 확인 또는 수정하면 이 정보를 이후 모든 블록에서 사용합니다.

**meta.json 생성**: OM 확인이 완료되면, 확정된 메타 정보를 **meta.json 파일로 즉시 저장**합니다. 이 파일이 이후 모든 Phase의 메타 정보 원천(single source of truth)입니다. LLM은 메타 정보가 필요할 때 대화 기억이 아닌 이 파일에서 읽습니다.

- **파일 경로**: 임시 폴더에 `{기업명}_meta_{fileTimestamp}.json` (세션 중에만 필요한 파일이므로 raw data 폴더에 남기지 않음)
  - Windows: `os.tmpdir()` (예: `C:/Users/GA/AppData/Local/Temp/`)
  - 경로 확인: `node -e "console.log(require('os').tmpdir())"`
- **파일 구조**:
```json
{
  "company": "한울회계법인",
  "course": "AI 엑셀 자동화",
  "round": "1차수",
  "instructor": "김태원",
  "date": "2026-04-02",
  "respondents": "25",
  "reportScope": "기업담당자용 + 강사용",
  "omName": "",
  "pluginStartTime": "2026-04-02T14:30:00.000Z",
  "htmlCreatedTime": "",
  "phase5Log": []
}
```
- **필드 규칙**:
  - 미확인 항목은 빈 문자열 `""` (null이 아님)
  - `omName`: 현재는 빈 문자열. 향후 OM 이름 수집 기능 추가 시 여기에 저장. GAS 전송 시 함께 포함됨
  - `pluginStartTime`: Phase 0 check-env.js가 출력한 `currentTime` 값 (ISO 형식, 코드가 찍은 정확한 시각)
  - `htmlCreatedTime`: Phase 4.5에서 HTML 생성 시 채움
  - `phase5Log`: Phase 5 수정 요청 시 누적 (최초 생성 시 빈 배열)
  - `reportScope`: "기업담당자용 + 강사용" / "강사용만" / "기업담당자용만"
- **이후 Phase에서의 활용**:
  - Phase 3: 블록 0(과정 개요) 작성 시 meta.json에서 읽음
  - Phase 4.5: HTML 헤더(기업명/과정명/부제) 작성 시 meta.json에서 읽음. generate-shell.js에 meta.json 경로를 전달하여 hidden 필드 자동 삽입
  - Phase 5: 수정 요청 원문을 phase5Log에 누적, 메타 수정 시 해당 필드도 갱신

### 블록 1) 객관식 — 스크립트 실행

> **⚠️ CONSTRAIN: LLM은 객관식 점수를 직접 계산하지 않습니다. 반드시 scripts/calculate.js를 실행합니다.**

`scripts/calculate.js`에 raw data의 파일 경로와 대상 시트명을 전달하여 실행합니다.

**⚠️ 경로 전달 규칙**: 스크립트 실행 시 파일 경로는 반드시 **절대경로**를 **따옴표(`"`)로 감싸서** 전달합니다. 대괄호`[]`, 한글, 공백, 괄호`()`가 포함된 파일명에서 쉘 해석 오류가 발생하기 때문입니다. 이 규칙은 calculate.js, verify.js 모두 동일합니다.

**⚠️ 스크립트 실행 경로 규칙**: 스크립트 실행 시 반드시 `cd ~/.claude/skills/report &&` 를 앞에 붙여서 플러그인 폴더에서 실행합니다. 이렇게 해야 사전 승인된 명령 패턴에 매칭되어 OM에게 불필요한 승인 요청이 나가지 않습니다.
```
예: cd ~/.claude/skills/report && node scripts/calculate.js "C:/Users/홍길동/Downloads/[기업명] 만족도 조사.xlsx"
```

**스크립트가 계산하는 것**:
- 일반 5점 척도 문항: 각 문항별 평균 점수 (소수점 1자리 반올림) + 긍정 비율 (4+5점 비율%, 기업담당자용)
- 양극단 척도 문항: 평균 점수 + 분포 비율 (긍정 비율 계산 안 함)
- 선택형 문항: 선택지별 응답자 수 + 비율(%)
- 정보성 문항 (`info`): 만족도 평가가 아닌 사실/상태 확인 문항 (교육 차수, 수강자 소속, 답변 적격성 등). 만족도 표에는 포함하지 않지만, `info` 섹션에 별도 보관. OM이 요청하면 활용 가능 (예: "수강자 소속 분포도 넣어줘", "소속별로 나눠서 보여줘")
- 총 응답자 수: 실제 응답 데이터 행 수 (헤더·빈 행 제외)

**양극단 척도 판별 규칙** — 아래 문항은 3점이 적절(가운데가 이상적)인 구조이므로 긍정 비율을 계산하지 않고 분포를 표시합니다:
- **난이도 인식** (난이도 정도): 1점(매우 쉬움) ~ 3점(적절) ~ 5점(매우 어려움)
- **강의 속도**: 1점(매우 느림) ~ 3점(적절) ~ 5점(매우 빠름)
- **교육 운영** (교육 시간): 1점(매우 짧다) ~ 3점(적절) ~ 5점(매우 길다)
- 판별 기준: 문항명에 "난이도 인식", "난이도 정도", "강의 속도", "교육 운영" 키워드가 포함되거나, 척도 설명에 "쉬움~어려움", "느림~빠름", "짧다~길다" 패턴이 있는 경우

**스크립트 결과 예시**:
```
[일반 5점 척도 문항]
  1. [커리큘럼] → 평균 4.1 / 5점, 긍정 비율 92%
  2. [인사이트] → 평균 4.3 / 5점, 긍정 비율 96%

[양극단 척도 문항]
  3-1. [난이도 인식] → 평균 3.2 / 5점, 분포: 쉬움(1~2점) 12% | 적절(3점) 58% | 어려움(4~5점) 30%
  4. [강의 속도] → 평균 3.1 / 5점, 분포: 느림(1~2점) 8% | 적절(3점) 65% | 빠름(4~5점) 27%
  9. [교육 운영] → 평균 2.8 / 5점, 분포: 짧음(1~2점) 15% | 적절(3점) 62% | 길음(4~5점) 23%

[선택형 문항]
  난이도 → 적절 58%(15명), 다소 어려움 23%(6명), ...

총 응답자: 40명
```

LLM은 이 결과를 **그대로 표로 배치**만 합니다. 수치를 수정하거나 재계산하지 않습니다.

**문항명 표기**: 대괄호`[]` 안의 카테고리명을 축약형으로 사용합니다.
(예: `2-1. [커리큘럼] 내가 기대한 강의 목표를 달성했다.` → `[커리큘럼]`)
대괄호가 없는 문항은 스크립트가 문항 텍스트에서 핵심 키워드를 추출합니다.

**기업담당자용 추가**: 긍정 비율 표시. HTML 표 헤더에 기준을 명시한다: 5점 척도는 **"긍정 비율 (4~5점)"**, 10점 척도는 **"긍정 비율 (7~10점)"**.
**선택형 문항**: 선택지별 응답자 수 + 비율(%) 표시

### 블록 2) 주관식 — 공통 (기업담당자용 / 강사용)

> **⚠️ CONSTRAIN: 요약문의 근거는 반드시 실제 원문에 기반합니다. 없는 내용을 지어내지 않습니다.**

**⚠️ 주관식 문항 성격 판단 (다문항 시 필수)**

주관식 문항이 여러 개일 때, 먼저 각 문항의 성격을 판단합니다:

- **포괄적/평가형 문항**: "자유롭게 작성해주세요", "좋았던 점", "아쉬웠던 점", "바라는 점" 등 열린 질문. 응답자가 어떤 내용이든 자유롭게 쓸 수 있는 문항.
  → **통합 유형 분류** (긍정/개선/추가 교육 니즈)

- **구체적/탐색형 문항**: "기억에 남는 것", "유익했던 교육 방식과 이유", "더 다뤘으면 하는 도구/기술", "현업 적용 시 어려움", "역량 향상 부분" 등 특정 관점을 물어보는 문항.
  → **문항별 답변 패턴 정리** (문항의 질문 맥락 안에서 핵심 키워드/패턴을 정리)

- **두 유형이 혼재하면 섹션을 나눠서** 포괄 의견(유형 분류)과 문항별 응답(패턴 정리)을 각각 보여줍니다.
- **주관식 문항이 1~2개이고 포괄적이면** 기존 방식(통합 유형 분류)으로만 처리합니다.

구체적/탐색형 문항의 경우, 짧은 응답("n8n", "실습" 등)도 문항 맥락 안에서는 의미 있으므로 "맥락 불분명"으로 제외하지 않습니다. 단, 문항과 무관하게 무의미한 응답(".", "없음" 등)은 여전히 제외합니다.

**처리 흐름:**

1. raw data에서 주관식 응답 전체를 추출합니다.
1-1. 주관식 문항이 여러 개면, 각 문항의 성격(포괄적/구체적)을 판단합니다.
2. **1차 필터링** (공통 — 기업+강사 모두 적용):
   - 무의미 응답: "...", ".", ",", "없습니다", "특이사항 없음", "없음"
   - 응원 메시지: "화이팅입니다", "감사합니다" 등 피드백이 아닌 응원
   - 판단 유보: "아직 모르겠습니다", "다음 강의까지 들어봐야 알 것 같습니다" 등 실질적 의견이 없는 응답
   - 맥락 불분명: 단어 1~2개로만 구성되어 긍정인지 부정인지, 요청인지 감상인지 판단할 수 없는 응답. **단, 구체적/탐색형 문항의 짧은 응답은 문항 맥락에서 의미가 충분하면 제외하지 않습니다** (예: "기억에 남는 것" 문항에 "n8n" → 유효).
3. **운영 사항 필터링** (공통):
   - **응답 단위로 개별 판단**합니다 (어떤 문항에서 왔는지와 무관하게 내용 기준).
   - **제거**: 기업 측 운영 사항 — 숙소, 식사, 시설, 주차, 안내 동선 등 우리(강사/OM)가 통제할 수 없는 것
   - **포함**: 교육 운영 사항 — 교안/교재 품질, 실습 툴/환경, 시간 배분, 일정 구성 등 다음 차수에 반영할 수 있는 것
   - 제외/포함 판단 사유를 블록 검토 탭의 필터링 표에 기록합니다.
4. **2차 필터링** (강사용만 — 강사 보호 목적):
   - 강사에 대한 직접적 비판이 과도한 경우 순화 또는 제외
   - 제외 시 사유를 기록하여 OM이 판단할 수 있도록 함
5. **포괄적 문항** → 유효 응답을 **긍정 의견 / 개선 의견 / 추가 교육 니즈** 세 큰 유형으로 분류한 뒤, 각각 세부 유형으로 나눕니다.
   - **긍정 의견**: 이번 교육에서 좋았던 점 (만족, 유익, 도움)
   - **개선 의견**: 이번 교육의 아쉬운 점 (속도, 난이도, 환경 문제 등)
   - **추가 교육 니즈**: **새로운 과정·심화 과정·반복 수강**에 대한 요청. 현재 교육과는 별개로 추가 교육을 받고 싶다는 의지가 드러나야 함. ("심화 교육 희망", "반복적인 교육이 필요", "다른 주제로도 해주세요" 등). 기업담당자에게는 후속 과정 제안 근거가 됨.
     - ⚠️ **개선 의견과의 구분**: 현재 교육의 내용·구성·방법을 바꿔달라는 요청(실무 데이터 활용, 사례 소개 확대, 사전 과제 제시, 실습 시간 확대 등)은 **개선 의견**이다. 현재 교육의 자료 보완 요청(참고 도서 추천, 유튜브/블로그 공유, 사후 자료 배포 등)도 **개선 의견**이다. "추가 교육 니즈"는 **별도의 교육 과정**(심화, 후속, 다른 주제)에 대한 요청이다.
5-1. **구체적/탐색형 문항** → 문항별로 답변 패턴을 정리합니다.
   - 문항의 질문을 제목으로, 유효 응답에서 핵심 키워드/패턴을 묶어서 보여줌
   - 비슷한 응답은 하나로 묶고 건수 표기 (예: "n8n 심화 (3건)")
   - 원문도 근거로 함께 제공
6. 포괄적 문항: 세부 유형마다 **대표 원문들을 종합한 요약문 1줄**을 작성합니다.
   구체적 문항: 문항별 **핵심 패턴 + 대표 원문**을 정리합니다.

**메일에 들어가는 형태 — 포괄적 문항만 있는 경우** (기업담당자용 / 강사용 동일 구조):
```
긍정 의견
- 실습 위주의 커리큘럼: 단순 이론 교육이 아닌 직접 AI 툴을 사용해보는 실습 위주의 강의 구성이 매우 유익했음.
- 강사의 전문성 및 준비성: 강사의 전달력이 뛰어나고 강의 자료 및 교재 준비가 철저하여 학습 몰입도가 높았음.

개선 의견
- 실습 시간 확보 필요: 교육 내용 대비 시간이 촉박하여 실습을 충분히 깊이 있게 따라가기에 아쉬움이 있었음.

추가 교육 니즈
- 심화/지속 교육: 한 번의 교육으로는 충분하지 않아 추가적인 심화 교육이나 지속적인 후속 과정을 희망함.
```

- **대표 케이스만** — 18개 원문을 전부 유형에 넣는 것이 아니라, 각 유형을 잘 보여주는 원문 몇 개를 근거로 삼아 요약문을 작성합니다.

**블록 검토 탭에서의 표시**:

### 블록 3) 운영진 의견 — 기업담당자용

> **이 영역은 LLM 초안 생성이 허용되는 유일한 영역입니다.**
> **⚠️ CONSTRAIN: 근거 없는 주장을 하지 않습니다. 반드시 출처를 명시합니다.**

**운영진 의견의 역할**:
운영진 의견은 설문 결과(블록 1·2)만으로는 전달되지 않는 **현장 맥락**을 보충하는 영역이다. 객관식 점수와 주관식 요약은 이미 위 블록에서 전달했으므로, 운영진 의견은 그 너머의 이야기를 한다.
- **기업담당자용**: 이번 교육이 현장에서 어떻게 운영되었는지 상황을 공유하고, 이슈가 있었으면 어떻게 대응할 것인지 개선 방향을 제시한다. 미흡한 부분이 있으면 소명하고 다음 차수에는 어떻게 하겠다는 방향을 안내한다.
- **강사용**: 잘된 부분을 구체적으로 칭찬하고, 교수법·진행 방식·준비 등 강사가 액션 취할 수 있는 보완 피드백을 전달한다.

**소스 활용 원칙**:
운영진 의견의 핵심 재료는 **강의관리시트**다. 현장 관찰, 이슈 대응, OM 소견 등 설문에 없는 정보가 여기에 있다.
- **핵심 소스: 강의관리시트** — "운영진 의견" 셀(현장 분위기, 긍정 피드백), "이슈" 셀(개선사항, 트러블슈팅)을 우선 탐색한다. "특이사항" 등의 키워드가 포함된 열/셀도 확인한다.
- **보조 참고: 주관식 패턴 → 객관식 점수** — 블록 1·2에서 이미 보여준 수치나 요약을 그대로 옮겨 적는 것은 운영진 의견이 아니라 반복이다. (구체적 수치 인용 금지 — "어렵게 느낀 수강생이 많았다" 같은 정성적 맥락은 허용) 다만, 수강생 추가 교육 니즈가 있었다는 정도의 안내는 가능하다. 특히 개선 의견에 대해서는, 주관식에서 나온 피드백을 토대로 **향후 대응 방향을 제시**하는 것이 운영진 의견의 역할이다. 피드백 자체를 반복하는 것이 아니라, 그 피드백에 대해 어떻게 할 것인지를 쓴다.
- **강의관리시트가 없는 경우**: 점수와 주관식 패턴만으로 보편적 수준의 초안을 생성한다. OM에게 "강의관리 시트가 없어 현장 맥락이 반영되지 않았습니다. 필요 시 수정해주세요."라고 안내한다.

**다중 시트 균형 규칙** (시트가 2개 이상일 때):
- 각 시트의 강의관리시트를 **모두 훑고**, 포함시킬 만한 내용이 있으면 빠뜨리지 않는다.
- 특정 시트에 인사이트가 더 많으면 그쪽 비중이 자연스럽게 커져도 괜찮다. 억지로 균등 배분하지 않는다.
- 다만 다른 시트에도 참고할 내용이 있으면 함께 언급하고, 어느 시트에서 나온 근거인지 구분한다. (예: "리더직군에서는 ~, 영업직군에서는 ~")
- 시트 간 내용이 유사하면 "리더·영업 공통으로" 등 범위를 명시하여 하나로 묶되, 특정 시트에만 있는 이슈/특이사항은 해당 시트를 명시한다.

**톤 규칙**:
- 현장 상황 공유 + 이슈 대응 방향 제시가 중심
- 만족도 점수를 "높다"고 자화자찬하지 않음 (4.2는 업계에서 무난한 수준)
- 강의관리 시트의 구체적 내용을 근거로 긍정 표현 (점수 자체보다 현장 관찰에 기반)
- 문항 텍스트를 해석하여 "새로운 관점을 제공했다" 등으로 풀어쓰지 않음
- 구체적 보완 방안 없이 한계만 언급하는 것은 삼감 (예: "비대면 한계가 있었습니다" → 어떻게 보완할지 없으면 빼기)
- 이슈를 언급할 때는 반드시 의미를 붙임 — "이런 이슈가 있었습니다"만으로 끝내지 않고, 향후 어떻게 할지(조율 제안 / 재발 방지 안내 / 단발성이라 우려 없음 등)를 함께 안내. 의미를 붙일 수 없으면 언급하지 않음
- "고객사", "고객사 교육팀" 등의 표현 삼감 — 수신인이 바로 그 기업 담당자이므로 어색함. "교육팀에서도 적극적으로 참여해주셔서" 등으로 자연스럽게 표현
- **추후 교육 영업 제안(후속 과정 제안, 교육 확대 논의 등)은 OM이 아닌 LD(기업교육 컨설턴트)의 역할**이므로 운영진 의견에서 주도적으로 제안하지 않는다. 다만 수강생으로부터 추가 교육 니즈가 있었다는 정도의 안내는 가능하다.

**OM 판단 지원 — 작성 근거 표** (근거 최소 2개):
강의관리 시트의 위치는 셀 병합이 있을 수 있으므로 행 번호 대신 구분점(`"운영진 의견"`, `"이슈 내용"` 등)으로 표기합니다.
raw data의 점수 근거(평균, 긍정 비율, 분포 등)는 위치를 `'시트명' (점수 계산 기준)` 형태로 표기합니다. "calculate.js 결과"라고 쓰지 않습니다 — OM은 스크립트명을 모릅니다.
```
| 구분 | 출처 | 위치 | 원문 | 반영 내용 |
|------|------|------|------|----------|
| 강사 평가 | 강의관리시트 | '강의요약(리더직군)' 중 "운영진 의견" | "전문성이 강의 전반에 잘 드러났음" | 긍정 피드백 |
| 접속 이슈 | 강의관리시트 | '강의요약(리더직군)' 중 "이슈 내용" | "사내망 이슈로 GPT 접속 지연" | 대응 안내 |
```

## Phase 4: Ralph 루프 — 자동 검증

> **모든 검증은 scripts/verify.js를 실행합니다. LLM이 직접 판단하지 않습니다.**

생성된 블록 데이터를 JSON 파일로 저장한 뒤, `scripts/verify.js`에 전달합니다.

**⚠️ JSON 저장 규칙**: 블록 결과 JSON은 반드시 **Write 도구**로 파일을 생성합니다. Bash의 heredoc이나 echo 리다이렉션을 사용하면 "expansion obfuscation" 보안 경고가 발생할 수 있습니다.
**파일 경로**: 임시 폴더에 `{기업명}_블록결과_{fileTimestamp}.json` (세션 중에만 필요한 파일이므로 raw data 폴더에 남기지 않음. `{fileTimestamp}`는 Phase 0 check-env.js가 출력한 값)

```
cd ~/.claude/skills/report && node scripts/verify.js <raw-data-경로> <블록-결과-JSON-경로> [강의관리시트-경로]
```

7개 항목을 검증합니다:

| # | 검증 항목 | 방법 |
|---|---|---|
| 1 | 응답자 수 일치 | 실제 응답 데이터 행 수 (헤더·빈 행·index 제외) == 블록 0의 응답인원 |
| 2 | 문항명 매칭 | 시트 헤더에서 추출한 축약 문항명 == 블록 1의 문항명 (문자열 비교) |
| 3 | 척도 표기 일치 | 각 문항의 점수와 만점 기준 (예: 4.1/5점)이 올바르게 표기되었는지 |
| 4 | 반올림 자릿수 통일 | 블록 1 내 모든 수치의 소수점 자릿수 일관성 (소수점 1자리 기준) |
| 5 | 주관식 원문 존재 확인 | 블록 2의 인용 문장이 raw data에 실제 존재하는지 (hallucination 탐지) |
| 6 | 운영진 의견 근거 원문 존재 | 근거 표의 원문이 실제 존재하는지 (source 필드에 따라 강의관리시트 또는 raw data 주관식에서 검색, 1차 소스에서 못 찾으면 다른 소스에서 fallback) |
| 7 | 점수 인용 정확도 | 운영진 의견 내 인용 수치 == calculate.js 계산 결과 |

**검증 결과 처리**:
- **전부 통과**: Phase 5로 진행
- **실패 항목 있음**: 해당 블록 자동 재생성 → 재검증 (최대 3회)
- **3회 재시도 후에도 실패**: 실패 항목과 원인을 OM에게 보고, 수동 확인 요청

## Phase 4.5: HTML 생성 — 블록 검토 + 메일 미리보기

Ralph 루프를 통과한 블록 데이터를 HTML 파일로 생성합니다. **블록 검토와 메일 미리보기를 처음부터 함께 제공합니다.**

**생성 방식 — 본문 분리 + 스크립트 조립**:

> **🚫 LLM은 CSS/JS를 절대 작성하지 않습니다.** `<style>`, `<script>`, `<!DOCTYPE>`, `<html>`, `<head>`, `<body>` 태그를 직접 쓰지 않습니다. 이 부분은 generate-shell.js가 자동으로 붙입니다.

1. LLM은 **가변 본문만** Write 도구로 임시 파일에 저장한다.
   - 파일 경로: 임시 폴더에 `{기업명}_본문_{fileTimestamp}.html` (세션 중에만 필요한 파일. `{fileTimestamp}`는 Phase 0 check-env.js 값)
   - 본문은 `<div class="container">` 안에 들어갈 내용만 작성 (CSS/JS/html/head/body 태그 없이)
   - generate-shell.js가 반환하는 `reviewHeader`와 `mailHeader`는 **LLM이 본문에 직접 포함**하여 작성한다 (블록 검토 탭 안 sub-tab 위에 reviewHeader, 메일 미리보기 탭 안 sub-tab 위에 mailHeader)
2. `scripts/generate-shell.js`의 **조립 모드**를 실행하여 최종 HTML을 생성한다. **meta.json 경로를 반드시 전달**한다.
   ```bash
   cd ~/.claude/skills/report && node scripts/generate-shell.js "기업명 과정명 만족도 리포트" "본문파일경로" "출력파일경로" "meta.json경로"
   ```
   → 스크립트가 head(CSS) + 본문 파일 내용 + hidden 필드(meta.json에서 자동 생성) + tail(JS)을 합쳐서 출력 파일로 저장.
   → meta.json의 `htmlCreatedTime`을 생성 시점으로 자동 갱신.
   → JSON `{ success: true, outputPath, reviewHeader, mailHeader }` 반환.

**⚠️ reviewHeader/mailHeader 가져오기**: 조립 모드 실행 전에, 먼저 모드 1로 실행하여 reviewHeader와 mailHeader 값을 확인한다.
```bash
cd ~/.claude/skills/report && node scripts/generate-shell.js "기업명 과정명 만족도 리포트"
```
→ JSON `{ head, tail, reviewHeader, mailHeader }` 반환. 이 중 **reviewHeader와 mailHeader만** 본문 작성 시 사용한다. head/tail은 무시한다 (조립 모드가 자동으로 붙임).

**HTML 헤더 영역 — generate-shell.js가 meta.json에서 자동 생성**:

> **🚫 LLM은 헤더 영역을 작성하지 않습니다.** 기업명, 과정명, 부제(교육일시/강사명/응답인원)는 generate-shell.js가 meta.json에서 읽어 자동 삽입합니다.

- **기업명**: `company-name` 클래스로 상단 강조 (예: `빙그레`)
- **과정명**: `h1`으로 기업명 바로 아래 큰 제목 (예: `BC AI 역량강화교육`)
- **부제**: 확인된 항목만 `|`로 구분하여 표시. 미확인(빈 문자열) 항목은 생략. (예: `2026.03.09~03.10 | 송유이 강사 | 26명 응답`)
- 강사명 뒤에 "강사" 호칭 자동 부착 (meta.json에 "강사" 포함 여부 확인 후 중복 방지)
- 응답인원 뒤에 "명 응답" 자동 부착

**HTML 구조** — 메인 탭 2개 + sub-tab:
```
[블록 검토]                    [메일 미리보기]
 객관식 | 주관식 | 운영진 의견     기업담당자용 | 강사용(송유이)
```

- **블록 검토 탭**: sub-tab으로 영역별(객관식/주관식/운영진 의견) 분리. 각 sub-tab 안에 생성 내용 + 근거 표/제외 표(드롭다운). 각 sub-tab 상단에 블록 검토 설명 문구(`review-description`) + 안내 배너를 배치.
- **메일 미리보기 탭**: sub-tab으로 기업담당자용/강사용 분리. 메일 제목 한 줄 + 본문 (HTML 헤더 미표시). 복사 버튼 2개: "본문 복사"(secondary) + "전체 복사 (제목+본문)"(primary).

**⚠️ 동적 구성 규칙 — "없으면 안 만든다"**:
- **주관식 유효 응답 0건**: 블록 검토의 "주관식" sub-tab 자체를 생성하지 않음. 메일에서도 주관식 섹션 제거.
- **운영진 의견 생략 조건**: 주관식 유효 응답 0건 + 강의관리 시트 없음이면, 근거가 부족하므로 운영진 의견 블록을 생략할 수 있음 (OM에게 "운영진 의견을 작성할 근거가 부족합니다. 생략할까요?" 확인).
- **리포트 범위에 따른 탭 생략**: Phase 1에서 "강사용만"이면 메일 미리보기의 "기업담당자용" sub-tab 미생성, "기업담당자용만"이면 "강사용" sub-tab 미생성.
- **메타 정보 부제**: 확인된 항목만 `|`로 구분하여 표시. 모든 항목 미확인 시 부제 행 자체를 생략.
- **sub-tab이 1개만 남는 경우**: sub-tab 바를 표시하지 않고 해당 내용만 바로 보여줌 (예: 강사용만이면 메일 미리보기에 sub-tab 바 없이 강사용 메일만 표시).

**안내 배치 규칙**:
- **블록 검토 탭 — sub-tab 위에 2개 블록 고정**:
  1. `review-description` (파란 배경): 전체 설명 ("메일에 들어갈 내용을 객관식/주관식/운영진 의견 영역별로...")
  2. `guide-banner` (노란 배경): 수정 안내 ("수정이 필요하면 Claude Code 대화창에서 말씀해주세요")
- **주관식/운영진 의견 sub-tab 안**: `context-notice` (배경 없음, 진한 파란 텍스트 + 볼드)로 맥락 설명. 주관식: 유형 분류 방식 안내. 운영진 의견: 강의관리 시트 유무에 따른 한계 안내.
- **운영진 의견 블록 검토에서 수신자 라벨 필수**: 운영진 의견 초안 위에 "기업담당자용" / "강사용 (강사명)" 라벨을 명시한다. 강사용만 생성하는 경우에도 "강사용 (김상모)" 라벨을 표시하여 어떤 수신자용인지 명확히 한다.
- **메일 미리보기 sub-tab**: 각 sub-tab(기업담당자용/강사용)에 `guide-banner`로 수정 안내 + 복사 버튼 안내. 복사 버튼은 **2개**: "본문 복사" + "전체 복사 (제목+본문)".

**메일 미리보기 탭 HTML 작성 규칙**:
- 기업담당자용과 강사용 메일은 완전히 별개 메일입니다. 절대 "(이하 동일)" 등으로 축약하지 않습니다.
- 메일 미리보기 내에서 운영진 의견 문단에는 `class="mail-opinion"`을 적용합니다. 주관식 의견과 운영진 의견 사이에 시각적 구분(상단 여백 + 구분선)이 생깁니다.
- **메일 미리보기에서는 HTML 헤더(기업명/과정명/부제)를 표시하지 않습니다.** 대신 메일 제목을 상단에 한 줄로 표시하고, 구분선 아래에 본문을 배치합니다. 헤더는 블록 검토 탭에서만 표시됩니다. (헤더와 메일 제목이 중복되어 정신없어 보이는 것 방지)
- **복사 버튼 2개**: "본문 복사" (보조 스타일, 연한 배경) + "전체 복사 (제목+본문)" (주요 스타일, 진한 배경). 버튼 라벨은 이 텍스트만 표시하고, 용도 설명(답장용/새 메일용 등)은 HTML에 넣지 않습니다.

**Hidden 필드 — generate-shell.js가 meta.json에서 자동 삽입**:

> **🚫 LLM은 hidden 필드를 작성하지 않습니다.** generate-shell.js가 meta.json을 읽어 본문 뒤에 자동으로 삽입합니다.

삽입되는 hidden 필드:

| 필드 ID | 값 | 출처 |
|---------|-----|------|
| `hidden-meta` | 메타 정보 JSON (company, course, date, instructor, respondents, omName) | meta.json |
| `hidden-start-time` | 플러그인 시작 시각 (ISO) | meta.json `pluginStartTime` |
| `hidden-html-created-time` | HTML 생성 시각 (ISO, 스크립트 실행 시점에 자동 기록) | generate-shell.js가 생성 |
| `hidden-phase5` | OM 수정요청 원문 (Phase 5 누적분) | meta.json `phase5Log` |

스크립트가 HTML entity 이스케이프(`"` → `&quot;`, `&` → `&amp;`, `'` → `&#39;`)를 자동 처리하므로 특수문자 걱정 불필요.

**HTML 파일명 규칙**: 파일명에 대괄호`[]`를 사용하지 않습니다. 일부 시스템에서 문제를 일으킬 수 있기 때문입니다. `기업명_과정명_만족도 리포트_{fileTimestamp}.html` 형태로 저장합니다. (`{fileTimestamp}`는 Phase 0 check-env.js가 출력한 값. **LLM이 시각을 추측하지 않습니다.**)
```
예: 동국제강그룹_AI Agent for HR 워크숍_만족도 리포트_20260401_1825.html
```

HTML 파일을 **raw data 파일과 같은 폴더**에 저장한 뒤, **브라우저에서 자동으로 엽니다**:
```bash
node -e "require('child_process').exec('start \"\" \"파일경로.html\"')"
```
⚠️ **경로 규칙 (자동 열기 전용)**: `node -e` + `exec('start ...')` 체인에서 백슬래시(`\`)는 이스케이프 체인(Bash→node→JS→cmd)을 거치며 소실됩니다. 파일 경로의 백슬래시를 **슬래시(`/`)로 변환**한 뒤 전달합니다. (예: `C:/Users/GA/Downloads/파일.html`)
⚠️ `start` 명령을 Bash에서 직접 실행하면 브라우저가 안 열릴 수 있습니다. 반드시 `node -e`로 감싸서 실행합니다.

그리고 대화창에 완료 안내를 출력합니다:

> "리포팅이 완료되었습니다. 브라우저에서 결과를 확인해주세요.
> - **블록 검토 탭**: 객관식/주관식/운영진 의견을 영역별로 확인 + 근거 자료
> - **메일 미리보기 탭**: 제목 + 본문 확인, "본문 복사" / "전체 복사" 버튼
>
> 수정이 필요하면 대화창에서 말씀해주세요. (예: "운영진 의견 톤 좀 더 부드럽게 해주세요")
>
> 📌 다른 과정의 리포팅을 진행하실 때는 새 터미널에서 Claude Code를 열어주세요. (1과정 1세션)

> md 파일이 생성되며, 아래 파일과 함께 AX팀에게 전달해주세요:

**⚠️ "마침" 입력을 요구하지 않습니다.** HTML이 열리면 파이프라인은 완료된 것입니다. OM이 수정 요청을 하면 그때 대응합니다.

## Phase 5: OM 수정 대응 (선택적)

OM이 대화창에서 수정을 요청하면 대응합니다. 요청이 없으면 이 단계는 건너뜁니다.

- OM이 수정 요청 → 어떤 블록인지 불명확하면 확인 질문:
  "객관식 / 주관식 / 운영진 의견 중 어떤 부분인가요?"
- **수정요청 원문 누적**: OM의 수정 요청 원문을 meta.json의 `phase5Log` 배열에 추가합니다. 블록 재생성 **전에** 먼저 meta.json을 업데이트합니다.
  ```
  meta.json의 phase5Log에 추가:
  "[HH:MM] 운영진 의견 톤 부드럽게 해주세요"
  ```
  - 각 항목은 `[시:분] 요청 원문` 형식
  - OM이 여러 번 수정 요청하면 배열에 계속 누적
  - 이 값은 HTML 재생성 시 generate-shell.js가 hidden 필드에 자동 반영
- **메타 정보 수정 요청 시**: OM이 "기업명 틀렸어, XX로 바꿔줘" 같은 메타 수정을 요청하면 meta.json의 해당 필드도 함께 갱신합니다.
- 해당 블록만 재생성
- **재생성 후 Ralph 루프 재실행 → HTML 재생성 → 브라우저 자동 열기 → 다시 완료 안내**
- **⚠️ 완료 안내는 HTML이 생성될 때마다 동일한 전체 안내를 출력합니다.** 수정 후 재생성이든 최초 생성이든 완료 안내를 항상 표시합니다.
- **⚠️ Phase 5 재생성 시 파일 덮어쓰기 규칙**: Phase 4~4.5에서 생성한 JSON/본문 HTML/최종 HTML의 파일 경로를 기억해두고, 재생성 시 해당 파일을 **Read 후 덮어쓰기**합니다. 새 타임스탬프로 파일을 만들지 않습니다. 원본 경로를 찾을 수 없으면 OM에게 "이전 HTML 파일 경로를 확인할 수 없습니다. 어디에 저장할까요?"라고 물어봅니다.

**info 데이터 활용**: calculate.js 결과의 `info` 섹션에 수강자 소속, 교육 차수 등 정보성 데이터가 포함될 수 있습니다. `info` 배열이 비어있지 않은 경우에만 아래와 같이 활용합니다.
  - OM이 별도 요청하지 않으면 만족도 표에 포함하지 않음 (기본 동작)
  - OM이 요청하면 활용 — 예: "수강자 소속 분포도 넣어줘" → info의 소속 데이터를 객관식 표 아래에 추가
  - `info`가 빈 배열이면 관련 안내를 하지 않음

**메일 제목 템플릿**:

```
기업담당자용: [패스트캠퍼스] {기업명} - {과정명} 만족도 결과 전달드립니다.
강사용:       [패스트캠퍼스] {강사명} 강사님께 - {기업명} {과정명} 만족도 결과 전달드립니다.
```
- `[패스트캠퍼스]`는 대괄호 포함 그대로 표기합니다.
- `{과정명}` 부분에 차수/그룹 정보가 있으면 포함합니다. (예: `AI 업무 혁신 교육 2회차 B그룹`)
- 다중 강사 시 강사용 메일은 강사별로 각각 생성하므로, 각 강사명이 제목에 들어갑니다.

```
예시:
```

**조립 규칙**:
- **객관식 표 형식**: 메일에서 객관식은 표 형태로 출력합니다. OM이 메일에 복사-붙여넣기 하기 편하도록 정리된 표를 제공합니다.
- **영역 간 시각적 구분**: 주관식 의견과 운영진 의견 사이에는 충분한 여백(빈 줄 2개 이상)을 넣어 영역이 분리되어 보이도록 합니다. HTML에서는 `mail-opinion` 클래스로 상단 패딩+얇은 구분선을 적용합니다. "[운영진 의견]" 같은 라벨을 명시적으로 쓸 필요는 없지만, 시각적으로 다른 영역임이 분명히 드러나야 합니다.
- **다중 시트(과정/차수)**: 여러 시트를 한 메일에 담을 때, 마무리 멘트는 메일 맨 끝에 1번만 씁니다. 시트마다 중복하지 않습니다. (참고: "리더직군"과 "영업직군"은 서로 다른 과정이고, 같은 과정을 반복 운영할 때 1차수/2차수로 구분합니다. 시트 구분명은 raw data 탭 이름과 메타 정보를 기반으로 표기합니다.)
- **다중 강사**: 강사별로 각각 강사용 메일을 생성합니다. 강사명이 포함된 문항은 해당 강사 점수만 표시합니다.

## Phase 6: 세션 로그 추출 (선택적)

OM이 "세션 로그 추출해줘"라고 요청하면 실행합니다. 요청이 없으면 이 단계는 건너뜁니다.

**⚠️ 세션 로그는 반드시 MD 파일로 생성합니다.** Claude Code 내부 세션 파일(JSONL)을 그대로 복사하지 않습니다. 사람이 읽을 수 있는 형태로 정리하여 MD 파일을 생성합니다.

**파일명**: `세션로그_{기업명}_만족도리포트_{YYYYMMDD}_{HHMM}.md`
**저장 위치**: raw data 파일과 같은 폴더

**MD 구조**:

```markdown
# 세션 로그: {기업명} 만족도 리포트

- 과정명: {과정명}
- 사용 파일: {raw data 파일명}, {강의관리시트 파일명 (있으면)}
```

**작성 규칙**:
- **축약/누락 금지**: 각 Phase에서 실제로 수행한 작업, 사용한 도구, 발생한 오류를 빠짐없이 기록합니다.

- **실행 명령어 기록**: 스크립트 실행 시 사용한 전체 명령어를 그대로 기록합니다 (경로 포함).
- **도구 호출 목록**: 각 Phase에서 사용한 도구(Read, Bash, Write)와 대상을 기록합니다. (예: "Read x3 — 교육 개요 탭, 강의요약 탭, raw data 시트1")
- **판단 근거 필수**: 이슈가 발생했을 때 단순히 "skip함"이 아니라, **왜** 그 판단을 했는지 근거를 명시합니다.
- **OM 상호작용 기록**: OM이 입력한 내용, OM에게 질문한 내용, OM의 응답을 모두 기록합니다.
- **도구 실행 결과 요약**: 스크립트(calculate.js, verify.js 등) 실행 결과의 핵심 내용을 요약합니다. 전체 JSON을 붙여넣지는 않되, 통과/실패 항목과 수치는 포함합니다.
- **개선 제안**: 파이프라인이나 플러그인의 구조적 문제가 발견되면 "개선 제안" 섹션에 기록합니다. 이 정보가 AX팀에게 전달됩니다.
